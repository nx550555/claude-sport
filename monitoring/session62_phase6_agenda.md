## 議題2: STEP 0.5 実施を毎回確実に保証する仕組み (Session_62 フェーズ3 完了時 追加)

**追加日**: 2026-04-27 (フェーズ3 完了時、ユーザー指示により追加記録)

**論点**:

- ユーザー側で「STEP 0.5 を実施したかどうか」を後から検証する手段が乏しい

**論点に含めるべき検討事項**:

フェーズ6 統合動作確認時に、以下のいずれか (または組合せ) を検討すべきか議題化:

- **(案A) 機械的チェッカー導入**: `scripts/verify_step05_executed.py` を作成し、結果反映 commit に対し直前 N時間以内の STEP 0.5 実行ログがあるか自動検証。`monitoring/health_check.py` から呼び出して WARN/ALERT 化
- **(案B) 構造化ログファイル新設**: `monitoring/step05_log.jsonl` を新設し、STEP 0.5 実行のたびに JSON Lines 形式で「実行日時 / 対象スポーツ / 検出件数 / ユーザー判断結果」を append。pending_actions.md 任意記述ではなく構造化された強制ログ
- **(案C) commit hook / pre-push hook**: 結果反映を含む commit に対し、commit message に `[STEP05:DONE]` または `[STEP05:N/A]` タグが含まれていることを必須化する Git hook を導入
- **(案D) records JSON の prediction_hit 更新時にフィールド追加**: `prediction_hit` を `true`/`false` 更新する際に、同じ entry に `step05_scanned_at` フィールドを必須化。未付与の prediction_hit 更新は health_check で検知
- **(案E) 出力フォーマット強制**: 結果反映タスクの最終報告フォーマットに「STEP 0.5 実施結果: N件検出 / 0件 (該当なし)」の行を必須化し、抜けたら自己チェック (CHECK-4) で気付ける構造

**判断必要時期**: フェーズ6 統合動作確認時。フェーズ4-5 完了後にユーザーと協議して、上記案 (またはその他案) のいずれを採用するか確定する。

## 議題4: タグ付与の粒度ガイドライン (Session_62 フェーズ4 ステージ3 完了時 追加)

**追加日**: 2026-04-27 (フェーズ4 ステージ3 完了時、4-3 情報源タグ義務化 実装に伴い記録)

**論点**:
CLAUDE.md【柱C】4-3 セクションで「タグは **主張ごと** に付与する」「1主張 = 1事実言明 or 1推論結論」「複数主張を含む段落は各主張の末尾にタグを付与」「句読点単位ではなく『論理的に独立した主張』単位で区切る」と規定したが、「論理的に独立した主張」の判定基準は記述者によってブレる可能性がある。同じ記述を複数の Claude セッションで再生成したとき、タグ付与単位が揺れて health_check の `miss_analysis_tag_compliance` 判定や CHECK-2 自己点検の結果が一貫しない懸念がある。

**具体的な解釈余地の例**:

```
Khachanov 6-2 6-3 で勝利 [FETCH:https://atptour.com/...]。Walton はクレー試合数 8試合のみ [FETCHER:tennis_elo]。
```

この記述を以下のいずれと解釈すべきかが現時点で曖昧:

- **解釈 A: 2主張**として扱う
  - 主張1: 「Khachanov 6-2 6-3 で勝利」(試合結果の事実) → `[FETCH:URL]`
  - 主張2: 「Walton はクレー試合数 8試合のみ」(選手スタッツの事実) → `[FETCHER:tennis_elo]`
  - 各主張の末尾にタグが付与されており、現行運用通り
- **解釈 B: 1主張のデータ列挙**として扱う
  - 「Khachanov の勝利 + Walton のクレー試合数」を一連の事実セットと見る
  - この場合タグは段落末1つで足りる、または各データ点に付くタグは「列挙の補足記号」扱い

解釈 A vs B のどちらを採用するかで以下が変わる:
- health_check の文単位タグ検出ロジック (`re.split(r"[。\n]+", ma)` で分解後の文ごと判定) の妥当性
- CHECK-2 自己点検時の「主張 1個に対しタグ 1個」要件の厳格度
- 文末タグなしの短い接続句 (例:「なお、」「ただし、」) の扱い

**追加で曖昧な境界例**:

- 推論連鎖: 「A [FETCH] のため B が起こり、C が想定される [INFER]」 → 「B が起こり」部分は中間推論で独立タグ要否?
- 否定主張: 「X ではない [FETCH:URL]」 → 否定言明も独立主張として扱うか?
- 程度副詞付き事実: 「serve in% は **やや低かった** 52% [FETCHER:tennis_serve]」 → 「やや低かった」評価語と「52%」数値は別主張か?
- 複合主語: 「A と B はともに敗退した [FETCH:URL]」 → 複数の対象に同じ事実が適用される場合、1主張か複数主張か?

**判断必要時期**: フェーズ6 統合動作確認時。粒度ガイドライン (例: 「動詞単位で区切る」「主語が変わったら必ず別主張」「数値+評価語は1主張」等) を協議し確定する。

**現時点のステータス**: **議題記録のみ。実装変更なし**。粒度ガイドライン確定までは現行運用 (記述者判断 + health_check の文単位検出 + 短い接続句 10文字未満 はカウント対象外) を継続する。

**論点**:
柱D サブセクション8 で「同一リーグの今後の試合は前回判断踏襲 (ユーザー再判断不要)」と規定したが、シーズン跨ぎや大会フォーマット変更時に再判断が必要となる可能性があり、踏襲の有効期限ルールが未確定。

**詳細検討事項**:

- **シーズン跨ぎでの再判断**:
  - (例) ATP 500 R1 を Session_X で「区分2」と判断 → ATP 2025-2026 シーズン終了後の 2026-2027 シーズン開幕時に踏襲継続するか / 再判断するか
  - シーズンごとに seeded chain 構造・bye ルール・出場選手層が変動するため、踏襲の有効性は無期限ではない可能性
- **リーグ内 round 階層ごとの踏襲範囲**:
  - (例) ATP 500 R1 の判断 (区分2) と R2 の判断 (区分1) は連動するか / 独立か
  - WTA 125K の R1 / R2 / R3 / SF / F で踏襲範囲をどこまで広げるか
- **大会フォーマット変更時の扱い**:
  - (例) ATP 250 が新規 250 シリーズに格上げされた場合、過去の「区分3」判断は無効化されるか
  - リーグ統合・分割 (例: Super League の structure 変更) 時の踏襲継続性
- **(案A) シーズン単位で踏襲リセット (シーズン跨ぎで毎回再判断)**
  - メリット: シーズンごとのリーグ実情を反映可能
  - デメリット: 運用負荷高
- **(案B) フォーマット変更時のみ再判断 (シーズン継続中は踏襲)**
  - メリット: 運用負荷軽減
  - デメリット: 同一フォーマットでも seeded chain 等の変動は反映されない
- **(案C) round 階層ごとに踏襲粒度を分離 (R1 と R2 以降を別判断)**
  - メリット: テニス R1 bye の特殊性に対応可能
  - デメリット: 踏襲ルールが複雑化

**判断必要時期**: フェーズ6 統合動作確認時。

**現時点のステータス**: **議題記録のみ。実装変更なし**。柱D サブセクション8 は現行通り「同一リーグの今後の試合は前回判断踏襲」で運用継続する。
